# 文件编码与编辑约束

本约束适用于 `energy` 项目内所有文件，用于避免后续在 Windows 环境修改代码、文档、配置、脚本、SQL、SQLite 数据库和 Git 提交时重复出现编码损坏、补丁失败、编译失败、二进制误提交和命令试错。

项目内所有文本文件统一限制为 UTF-8 without BOM。二进制文件不得按文本方式读取、重写或格式化；确需替换时只允许使用原工具生成或明确的二进制复制方式。

## 已确认问题

本次在修改 `ControlBatterySet.java` 时出现过两类编码问题：

1. 使用 PowerShell `Set-Content -Encoding UTF8` 写回 Java 文件后，文件被写入 UTF-8 BOM，`javac` 报非法字符 `\ufeff`。
2. 使用 PowerShell 重定向 `git show ... > file` 恢复文件后，文件被写成 UTF-16LE，后续补丁和 Java 编译均无法正确读取。

根因是 Windows PowerShell 5 的文本写入和重定向会按自身默认编码处理输出，不等同于保留 Git 或原文件字节。项目内文本文件必须保持 UTF-8 without BOM。

## 强制规则

1. 修改项目内任何文本文件时，优先使用 `apply_patch`。
2. 不使用 PowerShell `Set-Content`、`Out-File`、`>`、`>>` 重写项目内任何文本文件。
3. 如果 `apply_patch` 因中文乱码上下文匹配失败，只缩小补丁上下文到 ASCII 稳定片段，例如 import、方法名、调用语句；不得改用 PowerShell 文本重写绕过。
4. 需要恢复单个文件时，必须先确认该文件当前改动全部是本轮自己造成的，避免覆盖用户改动。
5. 恢复 Git 中的单个文件时，优先使用 `cmd /c "git show HEAD:path > path"`，不要用 PowerShell 重定向。
6. 涉及 Java 源码恢复或编码修复后，必须立即检查文件头字节，并执行编译或相关测试。
7. 发现新的、会导致重复试错的不明确操作时，必须先补充到本文档，再继续执行同类操作。
8. `03code/energy/sql/rysqlite3.db` 是二进制 SQLite 数据库文件。若它在测试或运行中出现变动，默认不要提交；除非用户明确要求，交由用户后续手动提交。

## SQLite 数据库文件

`03code/energy/sql/rysqlite3.db` 不按文本文件处理，不做格式化、编码转换、文本替换或 PowerShell 文本重写。

处理规则：

1. 审查工作树时，如果只有 `rysqlite3.db` 是非目标变更，默认视为运行或测试副产物，提交时先跳过。
2. 需要提交代码和文档时，使用显式路径 `git add`，不要使用 `git add -A` 把 `rysqlite3.db` 顺带加入。
3. 只有用户明确要求提交数据库，或本轮任务本身就是修改数据库内容时，才纳入提交。
4. 当前环境如果没有 `sqlite3`，不要反复尝试结构化 diff；直接说明无法本地解析二进制差异。

可行检查：

```powershell
git status --short
git diff --numstat -- 03code/energy/sql/rysqlite3.db
git diff --summary -- 03code/energy/sql/rysqlite3.db
```

提交代码和文档时显式排除数据库，例如：

```powershell
git add 01document/文件编码与编辑约束.md 01document/蓄电池采集整合计划.md
git add 03code/energy/src/main/java/com/shanhe/project/device/opt/service/ControlBattery.java
git add 03code/energy/src/main/java/com/shanhe/project/device/opt/service/ControlBatterySet.java
git add 03code/energy/src/main/java/com/shanhe/project/device/opt/service/impl/RestoreServiceImpl.java
git add 03code/energy/src/test/java/com/shanhe/project/device/opt/service/ControlBatterySetTest.java
```

## Git 操作

当前 Windows PowerShell 环境不支持 `&&` 作为命令连接符，不要使用 `git add -A && git commit ...`。

可行方式：

```powershell
git status --short
git diff --check
git add <明确文件路径>
git commit -m "message"
```

如果需要连续执行，使用分步命令，不依赖 shell 连接符。提交前必须确认：

1. `git diff --check` 无错误。
2. `git status --short` 中没有非本轮目标文件被误加入。
3. `rysqlite3.db` 只有在用户明确要求时才 staged。
4. 用户未要求提交时，不主动 commit。

## 可行命令

检查文件头：

```powershell
Format-Hex -Path 03code/energy/src/main/java/com/shanhe/project/device/opt/service/ControlBatterySet.java | Select-Object -First 1
```

正常 Java 源码开头应直接是 `package` 的字节：

```text
70 61 63 6B 61 67 65
```

不得出现：

```text
EF BB BF    # UTF-8 BOM
FF FE       # UTF-16LE
```

仅在确认当前文件改动全部是本轮自己造成时，用 `cmd` 保留 Git 输出字节恢复：

```powershell
cmd /c "git show HEAD:03code/energy/src/main/java/com/shanhe/project/device/opt/service/ControlBatterySet.java > 03code\energy\src\main\java\com\shanhe\project\device\opt\service\ControlBatterySet.java"
```

恢复后至少执行：

```powershell
mvn "-DskipTests=false" "-Dmaven.test.skip=false" "-Dtest=ControlBatterySetTest" test
```

或：

```powershell
mvn -DskipTests compile
```

## 禁止做法

以下做法会再次触发编码风险，禁止用于源码或文档重写：

```powershell
Set-Content -Path some.java -Value $text -Encoding UTF8
Out-File -FilePath some.java
git show HEAD:path > some.java
```

如确实需要批量机械替换，先拆成多个小 `apply_patch`；无法拆分时，必须先说明原因，并在写入后立刻做文件头检查和 Maven 编译验证。
