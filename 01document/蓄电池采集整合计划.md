# 蓄电池采集整合计划

## 1. 整合原则

本次整合按以下原则执行：

1. 先保留原有业务行为，再做结构收敛。
2. 先迁移采集层和协议层，再迁移状态机和调度逻辑。
3. 先桥接现有 `BatteryHandler`，避免一开始同时改采集和业务两层。
4. 每完成一批命令或一层能力，都要做一次编译校验和状态回填。

当前采用的路线不是“先完整重写再回头梳理”，而是“按层次逐步整合”，原因是这样更容易控制行为偏差和遗漏风险。

## 5. 命令对齐任务清单

以下命令是当前优先级最高的命令集合，后续按此顺序核对：

| 优先级 | 命令 | 作用 | 当前状态 |
|---|---|---|---|
| P0 | `02/82` | 电池实时数据 | 已按电池组号构造请求，待串口联调确认 |
| P0 | `06/86` | 电池组配置/规格信息 | 已按请求/应答对接入，待串口联调确认 |
| P0 | `07/87` | 电池组告警状态 | 已按电池组号构造请求，待串口联调确认 |
| P0 | `0B/8B` | 电池组参数 | 已拆分为 3 个告警等级轮询项，待串口联调确认 |
| P0 | `0D/8D` | 设备故障告警状态 | 已按电池组号构造请求，待串口联调确认 |
| P1 | `0E/8E` | 设备型号及软件版本 | 已确认请求/应答映射，待串口联调确认 |
| P1 | `EB` | 电池组工作模式 | 未纳入轮询，后续补 |
| P1 | `E9` | 屏蔽告警参数读取响应 | 未纳入轮询，后续补 |
| P2 | `88/89/8A/99/A8/F6/F8/F9` | 设置/清理/校正类 | 进入状态机迁移阶段再补 |

说明：

- 当前命令集合主要解决“先把采集主链路跑起来”。
- 设置类命令暂不优先进入主动轮询。
- 命令请求码和应答码要以原 C 代码的实际行为为准，不直接依赖枚举字面含义。
- 当前 Java 第一轮对齐结果：
  - `02 -> 82`
  - `06 -> 86`
  - `07 -> 87`
  - `0B -> 8B`
  - `0D -> 8D`
  - `0E -> 8E`
  - 其中 `0B/8B` 已按一般/异常/严重 3 个告警等级拆分轮询
- 接收缓冲当前已支持：
  - 半包保留尾包
  - 粘包连续解帧
  - 多帧场景保留未完成残留字节
- 运行态当前已支持：
  - 单命令超时自动重试
  - 接收缓冲上限裁剪
  - 非期望应答日志记录
- 当前已完成本地验证：
  - `BatteryCollectorFrameCodecTest` 已覆盖单帧、半包、粘包、噪声前缀场景
  - `mvn -q -Dtest=BatteryCollectorFrameCodecTest test` 通过
- 当前对原 C 主循环的结论：
  - `RS485_Command_Run` 是“先读外部请求，再处理内部待发送响应包，再做 watchdog/task flag”
  - `Response_Command_Package` 属于内部触发后回灌到 RS485 的待发送包，不等同于轮询请求应答
  - Java 当前处理策略应保留“收到帧统一分发”，但 only expected response 才清 pending

## 6. 风险点

### 风险 1：请求码和应答码对应关系判断错误

影响：

- 会出现串口能发、但设备不应答或应答不匹配。

控制措施：

- 每补一个命令，都对照原 C 代码函数和抓包结果确认一次。

### 风险 2：采集模块与旧串口链路冲突

影响：

- 两套链路同时占用串口，导致联调结果失真。

控制措施：

- 当前保持采集模块默认关闭。
- 未完成单通道验证前，不并行启用旧链路和新链路。

### 风险 3：桥接旧业务时隐藏数据假设

影响：

- 采集成功但业务侧误判字段含义。

控制措施：

- 当前先桥接旧 `BatteryHandler`，但后续必须补“标准结果对象”和字段对照表。
