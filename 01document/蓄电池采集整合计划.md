# 蓄电池采集整合计划

## 1. 调整后的总体方向

本次整合的目标不是把 `energy` 先适配原 M460 中间协议，而是把 M460 已承担的底层采集与上行控制能力收敛到 `energy` 内部模块中。`energy` 整合 M460 后，应成为直接面对下行采集模块、上行 Modbus 控制和原有网络上报的主系统。

新的协议定位如下：

| 协议文件 | 新定位 | 处理原则 |
|---|---|---|
| `01document/protocol/3_CM-02N 600节协议A0版_采集模块端_20231025.xlsx` | 6 路下行 RS485 采集蓄电池协议 | 作为第一优先级切入点，直接实现采集、电池单体数据解析、入库和电池组计算 |
| `01document/protocol/3_CM-02N 980节协议B0版_PC端_V2.0_20260415.xlsx` | 原 `energy` 与 M460 的 PC 端中间通讯协议 | 整合后不作为运行链路继续使用，仅保留实现方法和命令语义供内部控制逻辑复用 |
| `01document/protocol/3_CM-02N 980节协议B0版_网络端_V2.0_20250819.xlsx` | 原 `energy` 与 M460 的网络端中间通讯协议 | 整合后不作为新增中间协议继续使用，仅保留其中可复用的网络参数、控制语义和实现方法 |
| `01document/protocol/SH 980节协议A0版_Modbus_V4.8_20260105.xlsx` | 2 路上行 RS485 Modbus 控制协议 | 在下行采集和数据模型稳定后接入，作为外部系统控制入口 |
| 原 `energy` JSON/TCP | 网络上报协议 | 保留，不因 M460 整合而取消 |

核心原则：

1. 先从采集模块端协议入手，把 M460 底层采集能力整合为 `energy` 的独立模块。
2. 先保证电池单体数据、组数据、告警、故障、参数能够被采集、解析、入库和计算。
3. 按需建立新表或调整数据模型，不强行套用原有不合理业务结构。
4. 原 `energy` 业务只作为后续衔接对象，不作为底层整合的设计起点。
5. PC 端 / 网络端 980 节协议退出 `energy <-> M460` 中间通讯链路，但其中命令实现、状态机思路、参数含义可以作为内部方法复用。
6. 上行 Modbus 和 JSON/TCP 都应读取同一套 `energy` 内部标准数据模型，避免多套口径。

## 2. 目标架构

整合后的链路应调整为：

```text
下行采集模块/蓄电池
        |
        | 6 路 RS485
        v
energy 独立 battery collector 模块
        |
        | 解析、标准化、入库、计算
        v
energy 蓄电池标准数据模型
        |
        +--> 原 energy JSON/TCP 网络上报
        |
        +--> 2 路上行 RS485 Modbus 控制/查询
        |
        +--> 后续再衔接页面、告警、统计、运维业务
```

原链路中 `energy -> M460 -> 下行采集模块` 的中间协议不再作为目标架构的一部分。M460 的价值转为源码参考和行为参考。

## 3. 当前已有基础

仓库中已具备一部分独立采集模块基础：

- 已创建 `com.shanhe.project.collector.battery` 模块。
- 已具备 6 路下行采集配置模型。
- 已具备 RS485 帧编解码基础：
  - `START + address + command + length + payload + check + 0D`
  - 校验为地址、命令、长度、负载累加取低 8 位。
- 已具备通道运行态、轮询、超时、重试、接收缓冲处理。
- 已具备部分命令枚举、命令入口和 REST 调试入口。
- 已具备 `BatteryModuleFrameSummaryService`、`BatteryDeviceProtocolCode` 等 600 节模块端协议识别类。
- 已移除 600 节模块端帧默认桥接旧 `BatteryHandler` 的路径，后续入库、计算、980 兼容视图通过 `BatteryModuleFrameConsumer` 扩展。
- 已阻断 980 节 PC/网络端聚合命令直接下发到 600 节模块端通道；控制必须先映射为明确的 600 模块端命令，再进入采集通道显式命令队列。
- 已完成一次协议来源审查：当前自动轮询中使用的 `02/82`、`06/86`、`07/87`、`0B/8B`、`0D/8D`、`0E/8E` 属于 980 节 PC/网络端聚合协议，不属于 `3_CM-02N 600节协议A0版_采集模块端_20231025.xlsx` 的下行采集模块端命令。

这些已有实现可以保留，但后续优先级需要调整：从“桥接旧业务”转为“形成独立采集、存储和计算闭环”。

## 4. 当前项目审查结论

### 4.1 已收敛且应保持的边界

- 下行采集源头已明确为 `600节采集模块端` 协议，默认自动轮询只允许 `01/81`，不再从 980 节 PC/网络端聚合命令向下套。
- `BatteryDeviceProtocolCode` 是 600 节模块端协议码表；`BatteryAggregateCommandDefinition` 只表示旧 980 聚合语义，不能作为下行轮询命令来源。
- `BatteryModuleFrameDispatcher` + `BatteryModuleFrameConsumer` 已成为 600 节原始帧扩展点，入库、计算、兼容输出应作为显式 consumer 或标准模型服务实现。
- 模块地址轮询已按“全量发现 + 响应地址缓存”处理：默认保留 `1..246` 发现能力；稳定后单体地址可按响应缓存收敛，`246` 电流温度/组模块地址每轮固定追加采集，不因缓存收敛被跳过。
- 已确认 M460 中 `246` 组模块数据也是通过 `01/81` 轮询采集得到，M460 自行计算的是最高/最低/平均、压差、温差、SOC/SOH、告警等聚合结果；不能把 `246` 等同于完整组计算结果。
- 标准实时表已按 `dev_battery_report_log` 的 `monitor_data` / `pack_data` 兼容字段重新收敛：单体实时表保存旧 `BatteryMonitor` 主要字段，组实时表同时保存 `246` 原始组数据和轮询结束后的组计算结果，默认配置仍关闭，适合先联调验证再切流。

### 4.2 当前代码具备的扩展空间

- 下行采集：`BatteryCollectorService` 已支持多通道独立线程、地址范围、响应超时、重试、地址缓存、批次上下文和状态观测；缓存命中后的地址列表已固定追加 `246`。
- 协议解析：`BatteryModuleFrameDataParserService` 已覆盖 `81`、`91`、通用状态响应和自动设置地址响应；控制类命令已通过 `BatteryModuleControlCommandService` 构造，并由采集通道显式命令队列在自动轮询空档下发。
- 数据落库：`BatteryModuleRealtimeConsumer` 已支持批次内缓冲和批量 upsert，默认运行模型收敛为单体实时表、组实时表和缓存；历史表、独立告警事件表只作为可选扩展，不进入本轮建表。
- 组计算：`BatteryModuleGroupCalculationService` 已具备电压、温度、内阻、在线数和新鲜度计算入口，计算结果已写回 `battery_module_group_realtime`；SOC/SOH/容量等复杂算法等来源明确后再接入。
- 业务衔接：旧业务可以通过标准模型映射接入，但不应恢复 `600 原始帧 -> BatteryHandler -> 旧 980 业务` 的路径。

### 4.3 仍需控制的风险

- 真实串口未联调前，`responseTimeoutMs`、`requestGapMs`、`maxRetryCount`、地址缓存阈值仍需按现场总线响应速度校正。
- 当前标准实时表是“最新值”模型；历史趋势功能是否保留独立历史表需重新评估，优先考虑实时表 + 缓存承接查询，避免 SQLite 写入压力过大。
- 组计算已调整为每轮 `01/81` 轮询结束后触发；批次内按同一 `pollBatchNo` 的单体数据计算，并结合 `246` 组模块数据更新组实时结果。
- 旧 JSON/TCP 上报仍读取旧业务缓存/表，尚未切到新标准模型；切换前必须先做字段映射和对账。新单体/组实时表字段应尽量对齐 `dev_battery_report_log.monitor_data` / `pack_data`，后续逐步移除对 `dev_battery_report_log` 的实时查询依赖。`dev_battery_report_log` 可作为兼容历史记录继续写入，供管理后台历史查询、历史导出和未改造旧功能过渡使用。
- 上行 Modbus 暂不应提前实现写控制，需等待内部 600 模块端控制服务边界稳定。

### 4.4 本轮确认后的模型调整方向

- `battery_module_cell_realtime` 作为单体最新有效值表，按 `packNum + batNum` 唯一保存单体最新值；`batNum` 即 600 节单体模块地址，不再额外保存 `moduleAddress`。字段命名收敛为 `pack_num`、`bat_num`、`capacity`、`resistance_rage_slip`、`swollen_voltage`；`capacity`、`resistance_rage_slip` 不由默认 `01/81` 直接解析，解析单体基础数据后通过 `BatteryModuleCellCompatibilityFillService` 从缓存补充，取不到缓存则保持为空，不影响主采集流程。非成功响应不写入单体实时表，`success`、`responseFlag` 保留在原始帧日志中用于联调追溯。
- `battery_module_group_realtime` 作为组最新状态表，已合并原组实时和组计算语义，并按单体实时表思路收敛为 `pack_num` 唯一；调试字段和响应状态留在原始帧日志中。`246` 响应提供 `packCurrent`、`batteryPackFloatCurrent`、`batteryPackOuterVoltage`、`environmentTemperature1`、`environmentTemperature2` 等原始组数据；轮询结束后在同一表补齐最高/最低/平均电压、压差、最高/最低/平均温度、温差、最高/最低/平均内阻、SOC/SOH 预留、状态与采集质量等计算字段。统计定位字段使用 `*_bat_num` 口径。
- 单体与组实时表不再保留 `update_time`，`create_time` 表示最新一次采集写入时间，upsert 时同步刷新；数据新鲜度、最新单体时间和最新 `246` 组模块时间均以 `create_time` 为准。
- 单体与组实时表不再保留 `channel_name`、`port_name`，以配置中的全局唯一 `pack_num` 作为实时数据隔离键。原始帧日志仍保留 `channel_name`、`port_name`，用于串口联调和故障追踪。
- `battery_module_cell_history`、`battery_module_group_history` 不进入本轮迁移脚本。若后续页面/导出仍需要历史趋势，再基于采样周期、保留天数和 SQLite 写入能力决定是否启用；当前优先保证实时表和缓存。
- `battery_module_alarm_event` 不进入本轮迁移脚本。告警功能先预留入口，后续优先复用或扩展 `dev_alarm_log`，从标准实时表和组计算结果生成告警。
- `dev_battery_report_log` 后续应从“实时数据来源”降级为兼容/历史过渡表；独立模块可在轮询结束后按开关同步写入旧历史结构，管理后台历史查询可继续读取该表。页面实时态、JSON/TCP、告警、统计和 Modbus 的实时数据最终应直接读取单体实时表、组实时表和缓存。

## 8. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 继续按旧 `energy -> M460` 思路整合 | 会把已经废弃的中间协议固化进新系统 | 所有新任务以 `600节采集模块端` 协议和标准数据模型为起点 |
| 字段比例系数或状态位理解错误 | 入库数据和上报数据失真 | 每个 P0 命令必须按协议、源码、串口抓包三方核对 |
| 旧业务字段假设污染新模型 | 后续 Modbus、JSON、页面口径不一致 | 先建标准对象，再做旧业务映射 |
| 上行 Modbus 过早接入 | 控制命令无稳定数据基础，调试成本高 | 下行采集和标准模型稳定后再接入 |
| JSON/TCP 与 Modbus 数据来源不一致 | 外部系统看到的数据互相矛盾 | 两者都从同一标准模型读取 |
| 把 980 节聚合命令误当成 600 节下行采集命令 | 串口联调时请求对象和语义错误，导致设备不响应或解析错位 | 阶段 1 命令必须逐条标注协议来源；`02/82` 等重码必须按协议上下文区分 |
| 调试 REST 入口继续直发 980 聚合命令 | 绕过协议分层，重新形成伪 `energy -> M460` 链路 | 入口只保留兼容形态并返回拒绝结果，后续需新增明确的 600 模块端控制服务 |
| 联调配置缺项或误填导致采集线程反复异常 | 串口未真正进入稳定收发，日志被配置错误淹没 | 启动前校验通道名、串口名、设备地址；时间、缓冲、重试参数使用安全默认值兜底 |
| 每条单体数据都触发整组计算 | 600 节采集时数据库查询和写入被放大，影响低功耗设备稳定性 | 组级计算在每轮 `01/81` 地址轮询结束后触发，读取同批次单体和 `246` 组模块数据；不再按单条单体或单条组信息帧即时触发 |
| 将 `246` 当作计算结果而不是采集结果 | 会遗漏组电流、浮充电流、外组压、环境温度的真实来源，导致组计算触发和字段映射错误 | `246` 每轮作为必采地址处理；组计算在轮询结束后读取同批次单体和 `246` 采集结果 |
| 继续保留多张历史/计算表 | SQLite 写入压力和后续查询口径复杂度增加 | 优先实时表 + 缓存；历史和告警按业务确认后再启用或复用旧表 |

1. 6 路下行 RS485 与 6 组蓄电池是否固定一一对应，还是允许一条总线挂多组。
