# 蓄电池采集整合计划

更新时间：2026-05-15

## 1. 总体方向

本次整合不是恢复 `energy -> M460` 的 980 节中间协议链路，而是把 M460 已承担的底层采集、控制、计算和上行能力逐步收敛到 `energy` 内部。整合完成后，`energy` 应直接面对下行 600 节采集模块、内部蓄电池标准数据模型、JSON/TCP 兼容上报和后续上行 Modbus。

核心原则：

1. 下行采集以 `3_CM-02N 600节协议A0版_采集模块端_20231025.xlsx` 为运行协议来源。
2. 980 节 PC 端、网络端协议只作为旧 M460 行为参考和兼容语义参考，不再作为新增运行链路。
3. 页面、JSON/TCP、后续 Modbus 写控制必须调用内部服务，再由 600 模块端显式命令队列下发。
4. JSON/TCP 和 Modbus 最终读取同一套标准实时模型和缓存，避免多套数据口径。
5. `configId/devId/config_id` 可继续作为外部兼容字段和数据库字段保留，但内部业务不再把调用方传入的 `configId` 当作可变维度。

协议定位：

| 协议/链路 | 当前定位 | 处理原则 |
|---|---|---|
| 600 节采集模块端协议 | 下行采集主协议 | 直接实现采集、解析、入库、组计算和显式控制 |
| 980 节 PC 端协议 | 旧 `energy <-> M460` 中间协议 | 不恢复运行链路，只参考命令语义和状态机 |
| 980 节网络端协议 | 旧 JSON/TCP 聚合控制参考 | 不恢复任意透传，只按内部蓄电池服务逐项迁移 |
| SH 980 节 Modbus | 后续上行控制/查询协议 | 标准模型稳定后接入，写寄存器映射到内部控制服务 |
| 原 energy JSON/TCP | 外部兼容上报协议 | 保留，逐步从旧兼容历史切到标准实时模型 |

## 2. 目标架构

```text
下行采集模块/蓄电池
        |
        | 6 路 RS485，600 节采集模块端协议
        v
energy battery collector
        |
        | 解析、批次落库、轻量组计算、异步后处理
        v
energy 蓄电池标准实时模型
        |
        +--> 页面、告警、统计
        +--> JSON/TCP 兼容上报
        +--> 后续 Modbus RTU 查询与控制；Modbus TCP 是否纳入范围待确认
```

功能分层：

1. 通道配置：`BatteryPack.packNum` 对应电池组号，`BatteryPack.batSinSize` 对应该组期望单体数，6 路下行 RS485 固定映射 6 组蓄电池。
2. 自动采集：默认轮询 `01/81`。单体地址为 `1..245`，`246` 固定为组电流/温度模块地址。
3. 批次落库：每轮生成 `pollBatchNo`，批量写入单体实时表、组实时表，并执行轻量组计算。
4. 异步后处理：SOC/SOH、容量预测、告警上下文、兼容历史写入和缓存刷新不阻塞采集线程。
5. 控制命令：页面、JSON/TCP、后续 Modbus 写控制统一走内部控制服务和 600 模块端命令队列。
6. 对外读取：实时页面、告警、JSON/TCP、Modbus 逐步统一读取标准实时表和缓存；`dev_battery_report_log` 降级为兼容历史过渡表。

## 3. 当前已有基础

已具备并应保留的基础：

- `com.shanhe.project.collector.battery` 独立采集模块。
- 6 路下行采集配置模型、通道运行态、轮询、超时、重试和接收缓冲处理。
- RS485 帧基础：`START + address + command + length + payload + check + 0D`。
- `BatteryDeviceProtocolCode`、`BatteryModuleFrameSummaryService` 等 600 模块端识别和摘要能力。
- `BatteryModuleFrameDispatcher`、`BatteryModuleFrameConsumer` 作为 600 原始帧扩展点。
- `BatteryModuleRealtimeConsumer` 已支持批次缓冲、批量 upsert、单体实时表、组实时表和兼容历史过渡。
- `BatteryModuleGroupCalculationService` 已具备电压、温度、内阻、在线数、新鲜度等轻量组计算入口。
- 600 控制命令已通过 `BatteryModuleControlCommandService` 构造，并由采集通道显式命令队列发送。

已确认的采集边界：

- 自动轮询只允许 600 模块端 `01/81` 作为默认主命令。
- `02/82`、`06/86`、`07/87`、`0B/8B`、`0D/8D`、`0E/8E` 属于旧 980 聚合语义，不能作为 600 默认轮询命令来源。
- `246` 是组模块地址，不是组计算结果本身；它提供组电流、外组电压、环境温度等原始组数据。
- 每轮必须保留 `246` 采集；响应缓存可减少 `1..245` 的无效全量扫描，但不能跳过 `246`。

## 4. 标准数据模型方向

当前模型方向：

| 对象 | 定位 | 口径 |
|---|---|---|
| `battery_module_cell_realtime` | 单体最新有效值 | 以 `pack_num + bat_num` 唯一；字段向旧 `monitor_data` / `BatteryMonitor` 兼容 |
| `battery_module_group_realtime` | 组最新状态 | 以 `pack_num` 唯一；同时承载 `246` 原始组数据和轮询结束后的轻量组计算结果 |
| `dev_battery_report_log` | 兼容历史过渡 | 继续服务历史查询、导出和未改造旧功能；不作为长期实时主数据源 |
| `dev_alarm_log` | 标准告警落地 | 优先复用/扩展，不新增独立模块告警表 |
| `dev_config_attribute` | 蓄电池组属性/告警参数 | 继续配合 `dev_battery_pack` 使用，暂不改名 |
| `dev_template_attribute` | 电池组默认属性种子 | 保留，由迁移脚本维护 |

暂不做：

- 暂不新增独立 `battery_module_cell_history`、`battery_module_group_history`。
- 暂不删除数据库 `config_id` 字段。
- 暂不一次性迁移旧缓存 key。
- 暂不把 SOC/SOH/容量预测放入采集轮询线程。

## 7. 当前待办

### 7.1 低风险继续精简

1. 继续扫描内部 service 是否仍暴露无意义 `configId` 业务方法。
2. 内部调用点从 `xxx(Constants.DEFAULT_CONFIG_ID, packNum)` 继续收敛为 `xxx(packNum)`。
3. Controller、JSON/TCP handler 可保留 `configId/devId` 入参，但入口立刻归一为默认设备。
4. Mapper/XML、数据库字段和旧缓存 key 先保持兼容，只由 service 内部传默认值。
5. 日志和文档明确 `configId` 是兼容字段，不是内部业务维度。

### 7.2 采集通道后续收口

1. `BatteryCollectorChannelConfig.configId`、`BatteryModuleControlCommand.configId`、`BatteryPendingRequest.configId` 降级为日志/兼容写表字段。
2. 通道定位只按 `packNum` / `batteryGroup`，不再按 `configId + packNum`。
3. 继续保留外部 DTO 和原始帧日志中的默认 `configId`，便于兼容旧表和排查。

### 7.3 标准模型补缺

1. 页面实时态、JSON/TCP、告警、统计逐步改读实时表和缓存。
2. `dev_battery_report_log` 只保留历史查询、导出和未改造旧功能过渡。
3. SOC/SOH、核容值、备电时长、放电容量迁移到轮询外后处理服务。
4. 告警候选继续从实时表和组计算结果生成，最终落到 `dev_alarm_log`。

### 7.4 Modbus 后置接入

1. 当前只整理只读寄存器映射和空值/无效值口径；已存在 `BatteryModuleModbusReadMappingService` 和 `BatteryModuleModbusReadMappingServiceTest`，不启动 RS485 监听。
2. RTU 从站运行模型、站号、端口、并发访问、异常码、写寄存器权限需现场确认。
3. 写寄存器先映射到内部 600 控制服务草案，不提前固化运行链路。
4. Modbus TCP 是否纳入范围仍待确认。
