# 蓄电池采集整合计划

## 1. 调整后的总体方向

本次整合的目标不是把 `energy` 先适配原 M460 中间协议，而是把 M460 已承担的底层采集与上行控制能力收敛到 `energy` 内部模块中。`energy` 整合 M460 后，应成为直接面对下行采集模块、上行 Modbus 控制和原有网络上报的主系统。

新的协议定位如下：

| 协议文件 | 新定位 | 处理原则 |
|---|---|---|
| `01document/protocol/3_CM-02N 600节协议A0版_采集模块端_20231025.xlsx` | 6 路下行 RS485 采集蓄电池协议 | 作为第一优先级切入点，直接实现采集、电池单体数据解析、入库和电池组计算 |
| `01document/protocol/3_CM-02N 980节协议B0版_PC端_V2.0_20260415.xlsx` | 原 `energy` 与 M460 的 PC 端中间通讯协议 | 整合后不作为运行链路继续使用，仅保留实现方法和命令语义供内部控制逻辑复用 |
| `01document/protocol/3_CM-02N 980节协议B0版_网络端_V2.0_20250819.xlsx` | 原 `energy` 与 M460 的网络端中间通讯协议 | 整合后不作为新增中间协议继续使用，仅保留其中可复用的网络参数、控制语义和实现方法 |
| `01document/protocol/SH 980节协议A0版_Modbus_V4.8_20260105.xlsx` | 2 路上行 RS485 Modbus 控制协议 | 在下行采集和数据模型稳定后接入，作为外部系统控制入口 |
| 原 `energy` JSON/TCP | 网络上报协议 | 保留，不因 M460 整合而取消 |

核心原则：

1. 先从采集模块端协议入手，把 M460 底层采集能力整合为 `energy` 的独立模块。
2. 先保证电池单体数据、组数据、告警、故障、参数能够被采集、解析、入库和计算。
3. 按需建立新表或调整数据模型，不强行套用原有不合理业务结构。
4. 原 `energy` 业务只作为后续衔接对象，不作为底层整合的设计起点。
5. PC 端 / 网络端 980 节协议退出 `energy <-> M460` 中间通讯链路，但其中命令实现、状态机思路、参数含义可以作为内部方法复用。
6. 上行 Modbus 和 JSON/TCP 都应读取同一套 `energy` 内部标准数据模型，避免多套口径。

## 2. 目标架构

整合后的链路应调整为：

```text
下行采集模块/蓄电池
        |
        | 6 路 RS485
        v
energy 独立 battery collector 模块
        |
        | 解析、标准化、入库、计算
        v
energy 蓄电池标准数据模型
        |
        +--> 原 energy JSON/TCP 网络上报
        |
        +--> 2 路上行 RS485 Modbus 控制/查询
        |
        +--> 后续再衔接页面、告警、统计、运维业务
```

原链路中 `energy -> M460 -> 下行采集模块` 的中间协议不再作为目标架构的一部分。M460 的价值转为源码参考和行为参考。

## 3. 当前已有基础

仓库中已具备一部分独立采集模块基础：

- 已创建 `com.shanhe.project.collector.battery` 模块。
- 已具备 6 路下行采集配置模型。
- 已具备 RS485 帧编解码基础：
  - `START + address + command + length + payload + check + 0D`
  - 校验为地址、命令、长度、负载累加取低 8 位。
- 已具备通道运行态、轮询、超时、重试、接收缓冲处理。
- 已具备部分命令枚举、命令入口和 REST 调试入口。
- 已具备 `BatteryModuleFrameSummaryService`、`BatteryDeviceProtocolCode` 等 600 节模块端协议识别类。
- 已移除 600 节模块端帧默认桥接旧 `BatteryHandler` 的路径，后续入库、计算、980 兼容视图通过 `BatteryModuleFrameConsumer` 扩展。
- 已阻断 980 节 PC/网络端聚合命令直接下发到 600 节模块端通道；后续控制必须先实现明确的 600 模块端控制映射或内部命令服务。
- 已完成一次协议来源审查：当前自动轮询中使用的 `02/82`、`06/86`、`07/87`、`0B/8B`、`0D/8D`、`0E/8E` 属于 980 节 PC/网络端聚合协议，不属于 `3_CM-02N 600节协议A0版_采集模块端_20231025.xlsx` 的下行采集模块端命令。

这些已有实现可以保留，但后续优先级需要调整：从“桥接旧业务”转为“形成独立采集、存储和计算闭环”。

## 4. 当前项目审查结论

### 4.1 已收敛且应保持的边界

- 下行采集源头已明确为 `600节采集模块端` 协议，默认自动轮询只允许 `01/81`，不再从 980 节 PC/网络端聚合命令向下套。
- `BatteryDeviceProtocolCode` 是 600 节模块端协议码表；`BatteryAggregateCommandDefinition` 只表示旧 980 聚合语义，不能作为下行轮询命令来源。
- `BatteryModuleFrameDispatcher` + `BatteryModuleFrameConsumer` 已成为 600 节原始帧扩展点，入库、计算、兼容输出应作为显式 consumer 或标准模型服务实现。
- 模块地址轮询已按“全量发现 + 响应地址缓存”处理：默认保留 `1..246` 发现能力，稳定后只轮询有响应地址，避免长期扫描空地址。
- 标准实时表和组计算表已形成第一版闭环，但默认配置仍关闭，适合先联调验证再切流。

### 4.2 当前代码具备的扩展空间

- 下行采集：`BatteryCollectorService` 已支持多通道独立线程、地址范围、响应超时、重试、地址缓存、批次上下文和状态观测。
- 协议解析：`BatteryModuleFrameDataParserService` 已覆盖 `81`、`91`、通用状态响应和自动设置地址响应；后续控制类命令可继续在 600 节协议边界内补请求构造和结果处理。
- 数据落库：`BatteryModuleRealtimeConsumer` 已支持批次内缓冲和批量 upsert，后续历史表、告警事件表、兼容视图可以基于同一标准对象继续扩展。
- 组计算：`BatteryModuleGroupCalculationService` 已具备电压、温度、内阻、在线数和新鲜度计算入口，SOC/SOH/容量等复杂算法应等来源明确后再接入。
- 业务衔接：旧业务可以通过标准模型映射接入，但不应恢复 `600 原始帧 -> BatteryHandler -> 旧 980 业务` 的路径。

### 4.3 仍需控制的风险

- 真实串口未联调前，`responseTimeoutMs`、`requestGapMs`、`maxRetryCount`、地址缓存阈值仍需按现场总线响应速度校正。
- 当前标准实时表是“最新值”模型；历史趋势、清理策略、告警恢复事件还没有落地，不能直接替代全部旧业务表。
- 组计算当前按最新实时数据计算，虽然有 `pollBatchNo` 标识，但尚未强制只按同一批次单体计算；联调后需根据数据完整性决定是否改成批次口径。
- 旧 JSON/TCP 上报仍读取旧业务缓存/表，尚未切到新标准模型；切换前必须先做字段映射和双写/对账。
- 上行 Modbus 暂不应提前实现写控制，需等待内部 600 模块端控制服务边界稳定。

- [x] 定义采集原始帧表或日志表，用于联调追溯，已新增 `battery_module_frame_log` 与可选入库 consumer。
- [x] 定义电池单体实时数据表，至少覆盖单体编号、电压、内阻、温度、状态、采集时间、通道、组号，已新增 `battery_module_cell_realtime`。
- [x] 定义电池组实时数据表，至少覆盖组电压、组电流、环境温度、SOC/SOH 如有、在线状态、告警状态、采集时间，已新增 `battery_module_group_realtime` 第一版，SOC/SOH 后续由计算层补齐。
- [ ] 定义告警/故障快照或事件表。注意 `87/8D` 属于 980 聚合语义，不能直接等同于 600 节下行模块端命令；新表应从 600 解析结果、组计算结果和后续内部告警规则生成。
- [x] 建立 `采集帧 -> 标准对象` 的解析边界，已新增 600 节模块端 `81/91` 标准解析服务。
- [x] 建立 `标准对象 -> 入库` 的第一版服务边界：原始帧日志可保存解析后的标准字段，默认关闭，联调时通过 `rawFrameLogEnabled` 开启。

- [ ] 明确是否需要单体/组历史表双写；未确认前实时表只作为最新状态源，不承诺历史趋势能力。

### 阶段 3：电池组计算

目标：把原 M460 或旧业务中可复用的计算方法迁移到 `energy` 标准模型之上。

| `0B/8B` | 读取/上传电池组参数 | 参数来源需落到本地配置表或标准参数模型 |
| `0D/8D` | 获取/上传设备故障类告警状态 | 由标准故障模型生成 |
| `0E/8E` | 获取/上传设备型号及软件版本号 | 可作为设备信息兼容输出参考 |
| `39/E9`、`3B/EB` | 屏蔽参数 / 测试状态 | 可复用为内部控制状态查询，但不作为 600 节下行采集命令 |

PC 端 / 网络端 980 节协议中的设置、测试、均衡、网络参数等命令，不再作为中间通讯协议优先实现；只在阶段 5 或阶段 6 中作为内部命令服务复用。
当前代码层面已将这类命令入口改为兼容占位，不会直接写入 600 节下行模块端串口。

## 7. 数据表建议

具体表结构需要结合现有数据库和字段命名再落地，当前建议先按以下对象拆分：

| 对象 | 建议用途 |
|---|---|
| 采集通道配置 | 记录 6 路下行串口、组号、地址、启停、轮询参数 |
| 采集原始帧日志 | 保存关键原始帧、解析状态、错误信息，便于联调 |
| 电池单体实时数据 | 保存每节单体的最新电压、内阻、温度、状态 |
| 电池单体历史数据 | 按采样周期保存历史趋势数据 |
| 电池组实时数据 | 保存组电压、电流、温度、容量/SOC/SOH、统计指标 |
| 电池组历史数据 | 保存组级历史趋势数据 |
| 告警故障事件 | 保存告警、故障、恢复、确认等事件 |
| Modbus 寄存器映射 | 保存寄存器地址、读写属性、数据来源、写入动作 |

如果现有 `stat_battery_*`、`dev_battery_*` 等表能承接，应优先复用；如果字段语义不清或耦合旧业务，应新建标准表，再做旧业务映射。

## 8. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 继续按旧 `energy -> M460` 思路整合 | 会把已经废弃的中间协议固化进新系统 | 所有新任务以 `600节采集模块端` 协议和标准数据模型为起点 |
| 字段比例系数或状态位理解错误 | 入库数据和上报数据失真 | 每个 P0 命令必须按协议、源码、串口抓包三方核对 |
| 旧业务字段假设污染新模型 | 后续 Modbus、JSON、页面口径不一致 | 先建标准对象，再做旧业务映射 |
| 上行 Modbus 过早接入 | 控制命令无稳定数据基础，调试成本高 | 下行采集和标准模型稳定后再接入 |
| JSON/TCP 与 Modbus 数据来源不一致 | 外部系统看到的数据互相矛盾 | 两者都从同一标准模型读取 |
| 把 980 节聚合命令误当成 600 节下行采集命令 | 串口联调时请求对象和语义错误，导致设备不响应或解析错位 | 阶段 1 命令必须逐条标注协议来源；`02/82` 等重码必须按协议上下文区分 |
| 调试 REST 入口继续直发 980 聚合命令 | 绕过协议分层，重新形成伪 `energy -> M460` 链路 | 入口只保留兼容形态并返回拒绝结果，后续需新增明确的 600 模块端控制服务 |
| 联调配置缺项或误填导致采集线程反复异常 | 串口未真正进入稳定收发，日志被配置错误淹没 | 启动前校验通道名、串口名、设备地址；时间、缓冲、重试参数使用安全默认值兜底 |
| 每条单体数据都触发整组计算 | 600 节采集时数据库查询和写入被放大，影响低功耗设备稳定性 | 组级计算只在组信息帧入库后触发；后续如需更精细口径，再改为节流或定时计算 |

## 10. 下一步执行计划

### 10.1 最近一轮只做联调闭环

目标：不继续扩业务，先证明 600 节下行采集链路稳定。

- 准备数据库：执行 `03code/energy/sql/battery_collector_schema.sql`，只开启必要的 `rawFrameLogEnabled`、`realtimeDataEnabled`，`groupCalculationEnabled` 视是否有 `246` 组模块响应再开启。
- 单通道联调：只启用一个 `activeChannels`，打开 `debugEnabled` 和对应 `debugChannels`，确认 `01/81` 请求、响应地址匹配、超时、重试、缓存命中、缓存移除和重置全量发现。
- 数据核对：用抓包帧、原始帧日志、实时表三方核对 `81` 单体字段和 `246` 组字段比例系数。
- 运行观测：通过 `GET /collector/battery/status` 查看通道状态、当前地址、批次号、活跃地址数量、超时次数，确认日志没有被空地址扫描放大。
- 结论回填：联调完成后更新本计划，明确哪些配置值可作为默认值，哪些必须留给现场配置。

### 10.2 第二轮做 6 通道并发

目标：确认线程模型、串口资源、数据库批量写入和计算负载可承受。

- 启用 6 路通道，先不开启历史写入和旧业务映射。
- 观察每路响应耗时、整轮轮询耗时、缓存地址数量、批量 upsert 耗时和 SQLite 锁等待。
- 根据现场响应速度调整 `pollIntervalMs`、`responseTimeoutMs`、`requestGapMs`、`maxRetryCount`、`moduleAddressMissThreshold`。
- 判断是否需要把组计算从“组信息帧触发”改为“轮询批次结束触发”。

### 10.3 第三轮补标准模型缺口

目标：让标准模型可以承接后续 JSON/TCP、页面、告警和 Modbus。

- 设计单体历史、组历史、告警/故障事件表，明确写入频率和清理策略。
- 基于标准实时表和组计算表定义旧业务字段映射，不直接让旧业务读取 600 原始帧。
- 梳理容量、SOC、SOH、备电时长算法来源；没有明确来源时只保留字段，不做假算法。

### 10.4 第四轮再接原业务和上行

目标：在数据口径稳定后，逐步对外提供兼容能力。

- JSON/TCP 上报先做标准模型到旧上报字段的映射和对账，再切换数据源。
- 页面、告警、统计逐步读取标准模型，旧表需要保留时通过同步/视图适配。
- Modbus RTU 从站最后接入，读取同一标准模型；写寄存器先映射到内部 600 模块端控制服务，不恢复旧 `energy <-> M460` 中间链路。
