主要述求：
	1、当前蓄电池处理模式为：设备-->M460_Series_BSP_CMSIS_V3.00.002（底层采集/控制、modbus协议处理）-->energy（json上报、本地展示）。
	2、目标不是简单删除 M460 能力，而是将 M460 C 程序中的底层采集、控制、协议适配、硬件交互逻辑整合进 energy，最终由 energy 统一承担底层采集、本地业务、页面展示和上层平台通讯。
	3、M460 原有能力按 4 组蓄电池设计，整合到 energy 后扩展为 6 组蓄电池采集能力，下行通讯通道通过配置文件指定。
	4、energy 保留现有 json 上报能力，通过网口 TCP 与上层应用通讯；同时新增 modbus 协议与上层平台通讯能力，上行通讯通道通过配置文件指定。
	5、energy 部署在 linux，设备带屏通过浏览器展示。


一、目录结构
	01document：存放文档目录
		protocol：当前所用到的协议文档
	02sourceCode
		M460_Series_BSP_CMSIS_V3.00.002：c编写的蓄电池程序，与底层设备及energy交互，后续整合至energy独立模块
	03code
		energy：java项目，最终产出
	04other：备份目录，不用管
二、硬件
	1、六个下行RS485口，分别用于与6组蓄电池设备通讯。
	2、两个上行RS485口，用于与上层应用通讯（modbus协议）。
	3、两个网口，用于与上层应用进行 json 上报、TCP 通讯以及按需扩展的网络协议通讯。
	4、DIDO口如硬件存在，仅用于辅助开关量输入/输出、继电器/通道选择、声光告警或其他硬件控制；蓄电池设备数据通讯不通过 DIDO 承载。
三、软件
	1、系统：Debian GNU/Linux 11
	2、浏览器：wpe-webkit-fdo
	3、应用：jdk、springboot、sqLite、ehcache、logback

四、项目现状补充
	1、03code/energy 是 Spring Boot 单体应用，当前已包含设备通讯、协议解析、蓄电池业务处理、告警、历史数据、统计分析、巡检、平台同步、前端静态资源等模块。
	2、energy 现有通讯能力：
		（1）串口：已引入 jSerialComm，配置项位于 application-prod.yml 的 device-comm 节，当前默认 commType=port，串口名 ttyS9，波特率 115200。
		（2）TCP：已引入 Netty，具备 TCP 服务端/客户端处理能力，可用于网口侧设备通讯或上层平台连接。
		（3）协议解析：已有 CommServerDecoder、CRC16、累加和、256 校验、Modbus 协议类型配置、协议属性映射和通用字段转换。
	3、energy 现有蓄电池业务能力：
		（1）蓄电池实时数据处理：BatteryHandler、BatteryPackHandler 负责 82、86、87、8B、8D、8E 等蓄电池私有指令解析。
		（2）核心数据项：组压、组外压、充放电电流、浮充电流、环境温度、单体电压、单体内阻、单体温度、SOC、SOH、备电时长、内阻测试状态、氢气浓度、绝缘电阻、鼓包/漏液相关值等。
		（3）存储与缓存：SQLite 作为本地数据库，Ehcache 作为设备、告警、历史、电池状态等运行缓存。
		（4）业务闭环：包含实时记录、历史记录、告警判定、状态联动、内阻统计、容量预估、前端展示和 JSON 上报平台。
	4、协议资料位于 01document/protocol，当前包含 CM-02N 600 节、CM-02N 980 节 PC/网络端、SH 980 节 Modbus 等协议表，可作为直连改造的寄存器映射与验收依据。

五、目标架构
	1、改造前：
		底层设备/采集模块 -> M460 C 程序 -> energy -> 本地页面/平台 JSON 上报。
	2、改造后：
		底层蓄电池设备 -> RS485/串口 -> energy 内置采集与控制模块（整合 M460 逻辑） -> energy 业务处理 -> SQLite/Ehcache -> 本地浏览器页面/平台 JSON 上报/平台 Modbus 通讯。
	3、保留能力：
		（1）本地大屏浏览器展示。
		（2）平台 JSON 上报。
		（3）现有告警、历史、统计、配置、巡检等业务能力。
	4、新增/强化能力：
		（1）energy 整合 M460 中已验证的底层采集、控制、轮询、重试、超时、寄存器解析、硬件交互逻辑。
		（2）M460 原 4 组蓄电池能力扩展为 energy 的 6 组蓄电池采集能力，6 路下行串口/RS485分别与6组蓄电池设备通讯，通道关系通过配置文件指定。
		（3）energy 新增上层 Modbus RTU 通讯能力，通过2路上行RS485向上层平台提供实时数据、状态和告警，通道参数通过配置文件指定。
		（4）energy 保留现有 json 上报能力，通过网口 TCP 与上层应用通讯。
		（5）原 energy 与 M460 之间的中间通讯协议在整合后收敛为内部模块调用，形成统一的采集任务调度、协议适配、数据标准化、异常隔离和通道健康监测机制。

六、建议模块划分
	1、comm：通讯抽象层
		统一封装下行6路RS485采集通道、上行2路RS485 Modbus通道、Modbus TCP、上层平台 TCP/HTTP/MQTT 等通道，向上提供 send、receive、open、close、healthCheck 等能力。
	2、modbus：Modbus 协议层
		支持 RTU/TCP 报文编解码、CRC16、功能码 03/04/06/10、异常码处理、寄存器批量读取、写入和响应校验。
	3、battery-collector：蓄电池采集层
		负责 6 组蓄电池轮询、采集周期、超时重试、离线判定、采集任务优先级和互斥控制。
	4、battery-protocol：协议映射层
		根据 protocol 目录中的协议表维护寄存器地址、数据类型、倍率、小数位、符号位、单位、告警位定义和版本差异。
	5、battery-domain：业务标准化层
		将 Modbus 寄存器数据转换为 energy 现有 BatteryPack、BatteryMonitor、BatteryReportLog、告警与统计模型。
	6、report：上报层
		复用现有 ClientReportService、DataReportJob、HeartbeatJob 等 JSON 上报能力。

七、实施路线建议
	1、第一阶段：协议梳理与最小闭环验证
		输出寄存器映射表；选取 1 组蓄电池完成 RS485 直连读取；验证组参数、单体参数、告警状态、CRC 和异常响应。
	2、第二阶段：Java 采集模块建设
		实现 Modbus RTU 主站、通道管理、轮询任务、超时重试、报文日志和配置化寄存器映射。
	3、第三阶段：接入现有业务模型
		将采集结果写入现有电池缓存、实时记录、历史记录、统计和告警处理链路，保持前端接口尽量不变。
	4、第四阶段：多通道与整机稳定性
		支持6路下行RS485分别采集6组电池，支持2路上行RS485和2路网口与上层应用通讯；补齐通道离线、设备离线、采集失败恢复、日志留存和资源占用控制。
	5、第五阶段：现场验收
		完成 24/72 小时稳定性运行、断线恢复、断电重启、浏览器展示、平台上报一致性、采集准确率和告警准确性验证。

八、关键风险
	1、硬件接口风险：Java 在 Debian 上通过串口/RS485 与蓄电池设备通讯，需要确认串口设备节点、驱动权限、串口映射、波特率/校验位等参数；DIDO 仅需确认辅助控制场景和访问方式。
	2、实时性风险：原 M460 属于单片机实时采集，迁移到 Linux Java 后需通过线程优先级、轮询节拍、串口锁和队列隔离保证时序。
	3、协议差异风险：已有协议文件包含不同版本和不同端，必须明确现场设备实际版本，避免寄存器偏移、倍率、字节序不一致。
	4、兼容风险：前端和上报平台应尽量复用现有数据模型，避免改造扩大到页面、接口和平台协议。
	5、稳定性风险：SQLite 写入、Ehcache 缓存、串口读写和定时任务需要控制并发，避免采集高峰造成数据库锁或任务堆积。

九、推荐结论
	从现有代码和资料看，energy 已具备直接承接蓄电池采集的基础能力，包含串口通讯、TCP 通讯、协议校验、Modbus 类型配置、蓄电池解析、告警、历史、统计、SQLite 和页面展示。去除 M460 中间环节具备可行性，但建议采用“协议先固化、单组先验证、多组再扩展、接口尽量复用”的渐进方式实施。
