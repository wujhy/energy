# SH 980 Modbus 寄存器映射草案

本文基于 `SH 980节协议A0版_Modbus_V4.8_20260105.xlsx` 的“寄存器地址”页整理，仅作为后续阶段 5 的实现草案。当前不实现 Modbus RTU 从站/服务端运行链路，不提前固化写寄存器控制。

## 1. 接入原则

- Modbus 是 `energy` 整合 M460 后的上行 RS485 控制/查询接口，不是 `energy <-> M460` 中间协议。
- 读寄存器必须读取独立模块标准模型：`battery_module_cell_realtime`、`battery_module_group_realtime`、后续标准告警/故障模型。
- 写寄存器必须映射到内部控制服务，例如 `BatteryCollectorCommandService` / `BatteryModuleControlCommandService`，不能直接拼旧 980 命令下发到 600 下行总线。
- JSON/TCP、页面、告警、Modbus 的实时数据口径必须一致，优先从实时表和缓存读取。
- 在真实 600 下行采集、组计算、JSON/TCP 切源和控制命令响应落库稳定前，不实现 Modbus 写控制。

## 2. 协议基础

| 项目 | 草案结论 |
|---|---|
| 工作模式 | Modbus RTU，文档同时给出 Modbus TCP 示例；本项目先只考虑 2 路上行 RS485 Modbus RTU |
| 默认波特率 | 115200 BPS，需现场确认是否可配置 |
| 读功能码 | `0x03` |
| 写单寄存器 | `0x06` |
| 写多寄存器 | `0x10` |
| 异常响应 | 原功能码 + `0x80`；文档列出 `01` 无效功能码、`02` 无效数据地址、`03` 无效数据、`05` 收到命令、`06` 从机忙、`0E` 不支持的数据地址、`0F` 不支持的数据长度 |
| 地址规则 | 文档说明从机地址高 4 位为控制单元地址，低 4 位为电池组号；默认电池组 1..4 为 `0x11..0x14`。本项目有 6 组下行采集，组 5/6 的从机地址需联调确认 |
| 寄存器地址 | 文档提示请求中的寄存器地址需加 1 后等于响应中的寄存器地址；实现时需区分协议地址和文档参考号 |

## 4. 写寄存器映射草案

| 文档区域 | 内容 | 草案处理 |
|---|---|---|
| `404901` 起 | 蜂鸣器、系统状态、时间修改等系统状态寄存器 | 不默认实现；只有 energy 本机硬件或明确业务需要时再映射 |
| `400xxx` 参数设置 | 单压、组压、电流、温度、内阻、连接条、SOC、鼓包等告警阈值 | 暂缓；后续应映射到标准告警参数配置和 `dev_alarm_log` 规则，不直接写 600 模块 |
| `400421` 起 | 报警屏蔽 | 暂缓；后续映射到标准告警屏蔽/解除模型 |
| 测试/均衡/地址类控制 | 文档未在草案中逐项固化 | 后续写寄存器只允许调用内部控制服务，例如单体内阻测试、连接条测试、均衡、设置模块地址、自动编号等 |

写控制落地前必须先补齐：

- 命令互斥策略：同组同一时刻只能有一个显式控制流程。
- 响应结果模型：下发成功、响应成功、响应超时、设备拒绝需可查询。
- 与 JSON/TCP 控制一致：同一动作不能在 Modbus 和 JSON/TCP 中走不同业务路径。
- 权限与安全：维护类命令如地址设置、校准参数、清调试数据不能默认开放。

## 5. 后续实现顺序

1. 先实现只读寄存器映射服务，读取 `battery_module_cell_realtime` 和 `battery_module_group_realtime`，不启动 RS485 监听。
2. 用单测锁定寄存器地址、单体序号、倍率、偏置和空值策略。
3. 真实下行采集稳定后，再接入 2 路上行 RS485 Modbus RTU 从站。
4. 最后实现写寄存器到内部控制服务的映射。

## 6. 待确认问题

- 现场是否只使用 RTU，是否需要同时支持 Modbus TCP。
- 2 路上行 RS485 是完全等价双主访问，还是分别面向不同上位系统。
- 组 5/6 的从机地址是否沿用 `0x15/0x16`，还是文档只支持 1..4 组。
- 温度、电流、浮充电流等偏置值在现场上位系统中是否按文档执行。
- 告警位定义是否必须完全兼容 M460 原输出，还是允许按 energy 标准告警模型重新定义。
